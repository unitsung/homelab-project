# 文件管理策略：OMV 缺什么、App 补什么、OpenList 算不算多余

## 结论（先看）

1. **OMV 确实没有好用的现代文件管理**（Web 偏设置 NAS，不是 Drive 体验）。  
2. **App 端补齐文件管理是正确产品定位**——比在 App 里做 SMB/NFS 配置更有价值。  
3. **补齐方式仍需要一个「文件 API 后端」**：推荐继续用 **OpenList**，App 当手机/Mac 上的 Drive。  
4. OpenList 的「权限/存储配置」是 **服务端一次性（或很少改）的基建**，不是每天用的负担；日常浏览、搜、播才是 App 的活。  
5. **不要用 OMV RPC 当主文件管理后端**——能力只够列目录级别，不适合直链播放、网盘聚合、搜索。

---

## 1. OMV 为什么「文件管理不好用」

| OMV 强项 | OMV 弱项 |
|---|---|
| 挂盘、共享、用户、SMART | 没有 Drive 级 Web/App 体验 |
| SMB/NFS 给电脑用 | 手机原生挂 SMB 差（尤其 iOS） |
| 系统向管理 | 无统一「本地盘 + 网盘」入口 |
| FolderBrowser 等 RPC | 基本是选路径，不是完整文件产品 |

所以社区常见补法：

- **OpenList / AList**（统一列表 + 直链 + 多存储）  
- FileBrowser、Nextcloud、等  

你选 OpenList，是在补 **OMV 产品缺口**，不是叠床架屋。

---

## 2. 「App 补齐文件管理」实际指什么

```text
用户感知：          打开 MyHomelab → 文件 → 像网盘一样好用
底层必须有：        某个服务能 list / search / 出播放直链 / 鉴权

OMV 本体  ──共享/挂载──►  磁盘上的真实文件
                │
                ▼
         OpenList（文件 API + 权限模型）← 你在服务端挂什么目录是你的事
                │
                ▼
         MyHomelab App（只调 API：返回什么列什么，不绑死路径）
```

- **App 补的是体验**（列表、面包屑、图标、SenPlayer、错误态）。  
- **App 不配置、不写死**任何挂载路径；Immich/Jellyfin/OMV 同理，只消费 API。  
- **OpenList 补的是能力**（API、多存储、token、raw_url）。

---

## 3. OpenList「还要配权限」烦不烦？

| 配置类型 | 频率 | 谁做 | 备注 |
|---|---|---|---|
| 挂载 `tank/media` 等本地存储 | 很少 | 服务端一次 | 对上 OMV/ZFS 路径 |
| 挂载夸克等网盘 | 偶尔 | 服务端 | 和 QAS 生态匹配 |
| 用户 / token / 路径权限 | 很少 | 服务端 | 家庭用户通常 1 个管理员 token 就够 |
| 每天：进目录、搜片、外链播 | 高频 | **App** | 这才是产品价值 |

对照群晖：DSM 里也要先建共享文件夹和权限，**Drive App 并不会消灭服务端配置**，只是配置完之后日常只打开 App。

家庭场景可简化：

- 一个 OpenList 管理员 token 进 App Keychain  
- 存储只挂：`/影视` `/照片` `/音乐` `/下载` `/网盘`  
- 不必在 App 里做「权限编辑器」  

**App 里不要做 OpenList 存储源/权限管理 UI（一期）**——那是 Web 管理台的事；App 专注「已经授权范围内的浏览与使用」。

---

## 4. 若不用 OpenList，还有什么路？

| 方案 | 优点 | 缺点 | 是否推荐 |
|---|---|---|---|
| **OpenList + App UI** | 直链、搜索、网盘聚合、文档全 | 多一个服务；要配存储 | **推荐** |
| 直连 OMV `FolderBrowser` RPC | 少一个服务 | 无好用直链/播放链、无网盘、接口弱 | 不推荐主路径 |
| 手机直连 SMB/WebDAV | 「最原生」 | iOS 差、无统一体验、SenPlayer 链路别扭 | 不推荐 |
| FileBrowser 等 | 简单 | 弱于多存储/网盘/播放链 | 备选 |
| Nextcloud | 同步强 | 重，和「家庭影音入口」叠床架屋 | 除非已有 |
| App 自己 SSH/扫盘 | — | 危险、不可维护 | 禁止 |

**「App 补齐文件管理」≠「App 直连 OMV 文件系统」。**  
正确翻译是：**App 成为 OMV 存储的好用客户端，后端用 OpenList 暴露文件能力。**

---

## 5. 和夸克 / 网盘的关系

- 夸克转存：QAS（任务）  
- 转存落地后的文件：仍在盘或网盘路径上  
- 浏览/播放：OpenList 挂载同一路径  

App 文件页负责「看与播」，QAS 页负责「订与转」，职责清晰。

---

## 6. 产品表述（写给自己）

> OMV 负责盘与系统；OpenList 负责文件 API；MyHomelab 负责好用的文件管理 App 体验（以及照片/影视/下载/监控聚合）。

一句话：**不是「有了 OpenList 还要多配一层所以别做文件」；而是「正因为 OMV 文件体验差，才用 OpenList + App 把文件管理补齐」。**

## 7. 一期文件管理范围（与路线图对齐）

**做（App）：**

- 浏览、返回、面包屑（路径完全由 OpenList API 返回决定）  
- 列表元数据 + 图标/缩略图  
- 搜索（授权范围内）  
- 详情、复制直链、SenPlayer  
- 未配置 / 401 / 超时错误态  

**不做（App）：**

- OpenList 存储源与权限后台  
- 删除/移动/重命名（可二期）  
- SMB/NFS 配置（继续不做）  
- 用文件树替代 Immich（Immich 仍是独立照片库）  
- 替代 Jellyfin 的刮削/继续观看  

**服务端你要接受的一次性成本：**

- 部署 OpenList 并在服务端挂好你要暴露的目录  
- 发一个 token 给 App（App 不管理挂载）  

之后日常几乎只打开 App。  

备份 / rsync / ZFS 快照的监控见 [`storage-topology.md`](./storage-topology.md)，不放在文件 Tab 里做。
